# 常见夸大模式识别清单

自媒体和项目方常用的夸大手法。Phase 2/3 分析时逐条对照检查。每个模式包含识别特征、验证方法和判定标准。

## 模式1：API套壳（Shell Wrapping）

**含义**：号称"开源AI模型"，核心推理实际调用闭源API。

**识别特征**：
- 项目号称"开源模型"但代码中包含外部API调用
- 模型文件很小或不存在，推理走网络请求

**验证方法**：
```bash
# 搜索API调用关键词
grep -ri "openai\|anthropic\|claude\|gpt\|api_key\|requests.post\|httpx\|aiohttp" --include="*.py" .
# 搜索在线推理关键词
grep -ri "api.openai\|api.anthropic\|dashscope\|wenxin\|ernie\|qianfan" --include="*.py" .
```

**判定标准**：如果核心推理逻辑是API调用，则"开源模型"声称为假。项目本质是API客户端，不是模型。

## 模式2：评测注水（Benchmark Gaming）

**含义**：通过非标准评测手段制造高分假象。

**识别特征**：
- 自带评测脚本但包含"特殊处理"逻辑
- 评测数据集是自造的而非标准集
- 只报告最佳配置或最佳seed

**验证方法**：
1. 检查评测脚本中的数据加载逻辑（是否有过滤、采样）
2. 对比评测数据集与标准benchmark（文件名、样本数、内容）
3. 检查是否有 test set early stopping
4. 检查评测指标计算方式是否标准

**判定标准**：如果评测数据集非标准，或脚本包含选择性逻辑，性能声称不可信。

## 模式3：移花接木（Credit Hijacking）

**含义**：声称自研但核心能力来自第三方预训练模型。

**识别特征**：
- 声称"自研"或"从零训练"但依赖了预训练权重
- 项目号称"训练"但实际只做了微调或prompt工程

**验证方法**：
```bash
# 搜索预训练模型加载
grep -ri "from_pretrained\|load_model\|AutoModel\|pipeline\|ckpt\|checkpoint" --include="*.py" .
# 检查依赖
cat requirements.txt | grep -i "transformers\|sentence-transformers\|langchain\|llama-index"
```

**判定标准**：如果核心能力来自第三方预训练模型（如Llama、Qwen、GPT系列），"自研"或"从零训练"声称为假。

## 模式4：选择性报告（Cherry-picking）

**含义**：只展示有利结果，隐藏不利结果。

**识别特征**：
- 只展示好的案例或demo
- benchmark结果只报告有利指标
- demo只展示成功案例

**验证方法**：
1. 检查是否有完整的评测结果（所有子任务、所有指标）
2. 构造对抗样本或困难样本测试
3. 检查是否有失败案例被过滤

**判定标准**：如果只报告部分指标或部分数据集，性能声称需要打折。

## 模式5：概念偷换（Concept Switching）

**含义**：用特定条件下的表现冒充通用表现。

**典型手法**：
- 训练集表现 → 宣传为通用表现
- 特定领域表现 → 宣传为通用能力
- 实验室条件 → 宣传为生产可用
- 英文表现 → 宣传为多语言能力

**验证方法**：
1. 区分声称的适用范围与实际验证范围
2. 检查benchmark是否覆盖声称的适用范围
3. 在域外数据或不同语言上测试

**判定标准**：如果声称的适用范围大于实际验证范围，属于夸大。

## 模式6：参数造假（Parameter Fraud）

**含义**：声称的参数量与实际不符。

**识别特征**：
- 声称"百亿参数"但模型文件只有几百MB
- 配置文件中的参数量与声称不一致

**验证方法**：
1. 检查模型配置文件中的层数、隐藏维度、注意力头数
2. 手动计算参数量或用代码统计
3. 经验法则：FP16下1B参数约等于2GB文件大小

**判定标准**：如果实际参数量与声称不符，属于造假。

## 模式7：数据泄漏（Data Leakage）

**含义**：评测数据出现在训练数据中，导致虚高。

**识别特征**：
- 评测集与训练数据有重叠
- 评测数据与训练数据高度相似
- 模型对评测集的"记忆"程度异常高

**验证方法**：
1. 检查训练数据文件是否包含评测样本（文本相似度比对）
2. 计算训练集与测试集的n-gram重叠率
3. 构造全新样本测试（确保不在任何公开数据集中）

**判定标准**：如果存在数据泄漏，相关性能声称无效。

## 模式8：架构画饼（Vapor Architecture）

**含义**：架构图或论文描述了创新设计，但代码中未实现或被注释掉。

**识别特征**：
- 架构图中有模块但代码中找不到对应实现
- 代码中有大量 TODO 或 placeholder
- 关键模块被注释或跳过

**验证方法**：
1. 对照架构图/论文，逐个模块在代码中搜索实现
2. 检查关键函数是否只有 pass / TODO / NotImplementedError
3. 检查 forward 函数的实际执行路径

**判定标准**：如果声称的架构模块在代码中未实现，相关技术路线声称为假。
