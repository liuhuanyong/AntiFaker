# 声称分类法与验证方法

对开源项目的声称按类型分类，每类有对应的验证方法。Phase 1 提取声称时用于分类标注，Phase 3 验证时用于选择验证策略。

## 性能声称

**特征**：包含具体数字指标。

**典型话术**：
- "准确率达到98.5%"
- "推理延迟低于50ms"
- "支持10万QPS"
- "MMLU得分79.2"

**验证方法**：
1. 找到项目的评测脚本和评测数据集
2. 检查评测数据集是否为标准集（而非自造的"友好"数据集）
3. 检查评测脚本是否有特殊处理（如只评测有利样本、过滤困难样本）
4. 用标准benchmark重新跑一遍
5. 对比实测值与声称值

**常见问题**：
- 用非标准评测集（自造数据，当然分高）
- 评测集泄漏到训练集
- 只报告最好的seed或配置
- 延迟声称不含预处理或后处理时间
- 吞吐量声称基于理想输入长度

## 能力声称

**特征**：声称支持某类任务、语言、模态。

**典型话术**：
- "支持中英日韩四语言翻译"
- "能处理图文多模态输入"
- "支持代码生成和补全"

**验证方法**：
1. 检查模型配置文件中的支持列表
2. 检查 tokenizer / processor 是否支持声称的模态或语言
3. 检查推理代码中的输入处理逻辑
4. 实际构造输入测试

**常见问题**：
- 声称多语言但 tokenizer 词表不支持目标语言
- 声称多模态但图像分支是空壳或占位符
- 声称代码生成但实际调用了代码搜索API

## 技术路线声称

**特征**：声称使用了某种技术、算法、架构。

**典型话术**：
- "基于创新的多模态融合注意力机制"
- "采用自研的稀疏MoE架构"
- "使用对比学习进行预训练"

**验证方法**：
1. 定位模型定义代码（model.py / modeling_xxx.py / model_config.json）
2. 逐层检查架构是否与声称一致
3. 检查关键模块的实现是否真的是"创新"（还是标准实现换了个名字）
4. 对比论文引用的相关工作，确认是否为已有技术

**常见问题**：
- 标准Transformer换名叫"创新架构"
- 用了第三方库（如HuggingFace transformers）但声称自研
- 架构图画得花哨但代码实现是标准的

## 效率声称

**特征**：关于资源消耗的声称。

**典型话术**：
- "单卡GPU即可运行"
- "内存占用仅2GB"
- "模型大小仅350MB"

**验证方法**：
1. 检查模型参数量（checkpoint文件大小）
2. 检查推理代码的显存管理逻辑
3. 实际部署测量

**常见问题**：
- 声称的模型大小不含 tokenizer / vocoder 等附属组件
- "单卡运行"但需要A100 80GB
- 声称的内存占用不含推理时的KV cache

## 创新性声称

**特征**：声称首创、超越、颠覆。

**典型话术**：
- "业界首个XXX"
- "超越GPT-4"
- "颠覆传统方法"

**验证方法**：
1. 搜索是否有更早的同类工作（Google Scholar / arXiv / Papers with Code）
2. 检查"超越"的对比基线是否公平（同参数量？同数据量？同评测条件？）
3. 检查是否只在特定数据集上"超越"，在其他数据集上落后

**常见问题**：
- "超越GPT-4"但只在某个小数据集上
- "业界首个"但学术上早有类似工作
- 对比基线故意选弱的或用了不公平的评测条件
