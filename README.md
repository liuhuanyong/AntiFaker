# 开源项目声称验证（OSS Claim Verify）

## 概述

对开源项目的公开声称进行技术验证——提取声称、分析源码、跑评测、出报告。核心目标：将"打假"从每次从零开始的即兴工作，变为可复用的标准化流水线，大幅降低验证成本。

## 第一性原理

打假的根本矛盾是**声称成本约等于零，验证成本远大于零**。自媒体写一篇文章的成本是几分钟，验证方却要花几天读代码、跑实验。加速的关键不是让每一步快10%，而是改变验证成本的结构：

1. **流程标准化** — 不即兴发挥，按固定pipeline执行
2. **声称结构化** — 把散文式吹捧提取成可逐条验证的清单
3. **分析聚焦化** — 根据声称定向定位关键代码，不读全部源码
4. **报告模板化** — 填内容而非设计结构

## 工作流程

### Phase 1: 声称提取（Claim Extraction）

**目标**：从自媒体文章、项目README、论文中提取结构化声称清单。

**工具**：WebFetch（获取网页内容）、Read（读取本地文件）

**步骤**：
1. 收集信息源：自媒体文章URL、项目GitHub页面、README、论文（如有）
2. 用 WebFetch 获取内容
3. 提取声称，每条声称记录：
   - 编号（C1, C2, ...）
   - 类型：性能 / 能力 / 技术路线 / 效率 / 创新性（详见 `references/claim_taxonomy.md`）
   - 具体内容：原文怎么说的（引用原文）
   - 量化指标：如有具体数字（如"准确率98%"、"延迟<100ms"）
   - 可验证性：高 / 中 / 低
   - 验证方法：源码分析 / benchmark / 对比实验 / 文档检查
4. 输出：声称清单表格

**关键判断**：区分"可验证的硬声称"（有具体指标、可复现）和"模糊的软声称"（如"业界领先"）。优先验证硬声称，软声称标注为营销话术即可。

### Phase 2: 源码取证（Source Forensics）

**目标**：获取项目源码，根据声称清单定向分析关键实现。

**工具**：Bash（git clone）、Agent with Explore subagent（代码探索）、Grep / Glob（代码搜索）

**步骤**：
1. 获取源码：
   - `git clone <repo_url> /tmp/oss-verify-<project-name>`
   - 或用 WebFetch 查看 GitHub 页面（快速浏览场景）
2. 项目架构速览：
   - 用 Agent (Explore, thoroughness: medium) 生成项目结构摘要
   - 识别核心模块、入口文件、配置文件、评测脚本
3. 定向取证（根据声称清单逐条定位）：
   - 声称编号 → 文件路径 → 行号 → 代码片段
   - 重点关注：模型架构定义、训练流程、推理逻辑、评测脚本、数据处理
4. 依赖与外部服务检查（**最高优先级**）：
   - 检查是否依赖闭源API（搜索 openai / anthropic / claude / api_key / requests.post）
   - 检查是否包装了已有模型而非自研（搜索 from_pretrained / load_model / pipeline）
   - 检查 requirements.txt / setup.py / pyproject.toml 中的关键依赖

**关键判断**：很多"开源项目"实际上是"开源了一个壳，核心靠闭源API"。检查API调用是第一步，往往一步就能定性。

### Phase 3: 实现验证（Implementation Verification）

**目标**：逐条对比声称与实际实现，判定一致性。

**步骤**：
1. 对每条声称，填写验证结果：
   - ✅ 一致：实现与声称相符
   - ⚠️ 部分一致：核心属实但有夸大（如声称自研但实际用了第三方库）
   - ❌ 不一致：实现与声称矛盾
   - ❓ 无法验证：需要运行才能确认
2. 记录证据：代码片段、配置、依赖说明
3. 识别夸大模式（详见 `references/exaggeration_patterns.md`）：
   - API套壳、评测注水、移花接木、选择性报告、概念偷换、参数造假、数据泄漏

**关键判断**：验证的核心不是"代码能不能跑"，而是"代码做的事情和声称做的事情是不是同一件事"。

### Phase 4: 量化评测（Empirical Evaluation）

**目标**：用标准benchmark验证性能声称，给出量化数据。

**步骤**：
1. 评测设计：
   - 根据声称类型选择benchmark（NLP → MMLU/C-Eval/CMMLU，CV → ImageNet/COCO，代码 → HumanEval/MBPP，Agent → AgentBench）
   - 确定评测指标：准确率 / F1 / 延迟 / 吞吐量
   - 确定对比基线：与声称的数字对比，与已知开源模型对比
2. 环境搭建：
   - 按项目文档安装依赖
   - 记录环境信息：GPU型号、CUDA版本、Python版本、依赖版本
   - 如环境搭建失败，记录失败原因（**这本身就是一个重要发现**）
3. 运行评测：
   - 优先使用项目自带的评测脚本（如有），同时检查脚本逻辑是否公平
   - 补充标准benchmark评测
   - 记录原始结果数据
4. 结果对比：实测 vs 声称、实测 vs 已知基线模型

**关键判断**：
- 项目连环境都搭不起来，本身就是"不可复现"的证据
- 项目自带评测脚本时，检查是否有"偷分"逻辑（测试集泄漏、选择性评测）

### Phase 5: 报告生成（Report Generation）

**目标**：输出结构化技术验证报告。

**工具**：`assets/report_template.html`（报告模板）

**步骤**：
1. 读取 `assets/report_template.html` 作为模板
2. 填充以下内容：
   - 项目概览：名称、仓库地址、Star数、声称来源
   - 声称清单：完整表格
   - 逐条验证结果：声称-证据-判定矩阵
   - 量化评测数据：实测 vs 声称对比表
   - 结论：总体可信度评级（可信 / 部分可信 / 不可信 / 无法验证）
   - 建议：对决策层的建议
3. 输出为单文件HTML，保存到工作目录

**报告原则**：
- 论据推结论，不结论先行
- 每个判定都有代码或数据支撑
- 区分"确认造假"和"无法验证" — 后者同样是重要发现
- 给决策层看的结论要简明：可信 / 不可信 / 部分可信 + 一句话理由

## 快速模式（30分钟初判）

当时间紧迫、只需快速判断时，执行最小化验证：

1. **Phase 2 的依赖检查**（10分钟）— 搜索API调用和第三方模型加载
2. **验证最核心的1-2条声称**（15分钟）— 定位关键代码，初步判断
3. **输出简短结论**（5分钟）— 一句话定性 + 关键证据

适用场景：领导临时问"这个项目靠谱吗"，需要短时间内给初步判断。
